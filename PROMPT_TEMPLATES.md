# 简擎 AI Prompt 模版

本文提供 7 套可直接复用的 Prompt 模版，用于提升简擎项目后续 AI 辅助开发的一致性与可追溯性。

## 使用原则

- 硬规则以 `AGENTS.md` 为准。
- 执行上下文以两端三文件为准：
  - `jianqing-backend/task_plan.md`
  - `jianqing-backend/findings.md`
  - `jianqing-backend/progress.md`
  - `jianqing-admin-web/task_plan.md`
  - `jianqing-admin-web/findings.md`
  - `jianqing-admin-web/progress.md`
- Prompt 中只写本轮任务目标，不重复粘贴大段历史信息。

## 模版选择速查表

- 日常迭代、小需求、小改动：用「模版一：日常开发（简版）」
- 路由/API 变更、方法约束收敛：用「模版二：接口改造（严格版）」
- 发布前总检查、出发布说明前：用「模版三：发布前检查（Checklist 驱动）」
- 缺陷处理、线上问题、联调异常：用「模版四：BUG 修复（复现与回归优先）」
- 新功能开工前、需先评审再落地：用「模版五：需求评审 / 方案设计」
- DDL、字段变更、增量修复脚本：用「模版六：数据库变更（DDL / 迁移脚本）」
- 菜单/按钮/角色/越权修复：用「模版七：权限模型变更（RBAC 专用）」

## 模版一：日常开发（简版）

```text
你现在在简擎工作区开发，请按以下流程执行：

1) 先读取并遵循：
- AGENTS.md
- 后端三文件：task_plan.md / findings.md / progress.md
- 前端三文件：task_plan.md / findings.md / progress.md

2) 必须遵守：
- 后端遵循阿里巴巴 Java 开发规范
- 实现优先简单直接、可读、可维护
- 关键分支/复杂流程/边界处理写必要注释
- 前后端接口方法仅允许 GET/POST，不使用 PUT/DELETE
- 涉及跨前后端改动时，同步更新两端 planning 文件

3) 本次任务：
<在这里写本轮目标>

4) 输出要求：
- 先给“当前状态 + 下一步最小可执行任务（1~3项）”
- 再实施改动
- 最后给：变更清单、验证结果、下一步建议
```

## 模版二：接口改造（严格版）

```text
你现在负责简擎接口改造任务。请严格执行：

1) 启动阶段
- 读取 AGENTS.md + 前后端三文件
- 全量扫描接口定义与前端调用（先查后改）
- 先输出改造清单（按“后端路由/前端调用/文档”分组）

2) 约束
- 仅允许 GET/POST 接口方法
- 禁止新增 PUT/DELETE/PATCH
- 命名语义清晰：查询用 GET，写操作用 POST（可用 /update /delete /assign 后缀）
- 改动要保持最小闭环，不做无关重构

3) 实施顺序
- 先改后端控制器与相关 DTO/服务调用
- 再改前端 API 调用层
- 再改文档与示例（如 README/API_EXAMPLES）
- 最后做全量残留扫描，确保无 PUT/DELETE

4) 验证要求
- 后端：mvn -DskipTests compile && mvn test
- 前端：npm run build
- 扫描：后端无 @PutMapping/@DeleteMapping；前端无 http.put/http.delete

5) 交付格式
- 风险点
- 修改文件清单
- 验证结果
- 对 planning 文件的同步说明
```

## 模版三：发布前检查（Checklist 驱动）

```text
你现在执行简擎发布前检查。请按 checklist 逐项完成并回填结果：

[规范与一致性]
- AGENTS.md 规则是否全部满足
- 前后端 planning 文件是否与当前代码一致
- 是否仍满足“接口仅 GET/POST”约束

[后端质量]
- mvn -DskipTests compile
- mvn test
- mvn checkstyle:check
- README / CONTRIBUTING / API_EXAMPLES 是否与代码一致

[前端质量]
- npm run build
- 关键页面（登录、系统管理、审计日志）流程是否正常
- 权限显隐是否符合预期

[交付输出]
- 发布范围（v0.1）
- 已知限制
- v0.2 下一步计划（1~3项）

输出要求：
- 用“通过/不通过 + 证据”格式逐项给出结果
- 若存在不通过项，给最小修复方案
```

---

## 推荐用法

- 日常任务优先使用“模版一”。
- 涉及路由/API 变更时使用“模版二”。
- 准备打 tag 或发布说明前使用“模版三”。

## 模版四：BUG 修复（复现与回归优先）

```text
你现在负责简擎 BUG 修复任务。请严格执行：

1) 先读取上下文
- AGENTS.md
- 前后端三文件（task_plan.md / findings.md / progress.md）

2) 先复现，后修复
- 先明确：现象、触发条件、预期结果、实际结果
- 输出最小复现步骤（输入、操作、结果）
- 未复现前不要盲改

3) 修复约束
- 只修根因，不做无关重构
- 优先最小改动闭环
- 保持代码简单可读
- 关键分支与边界处理写必要注释
- 接口方法仅允许 GET/POST，不新增 PUT/DELETE

4) 实施顺序
- 定位影响范围（后端/前端/接口/文档）
- 给出 1~2 个修复方案并选择最小方案
- 实施修改
- 执行回归验证（覆盖“原问题 + 相邻功能”）

5) 验证要求
- 后端变更：mvn -DskipTests compile && mvn test
- 前端变更：npm run build
- 接口变更：额外执行方法约束扫描（确认无 PUT/DELETE 残留）

6) 输出格式
- 问题根因
- 修改清单（文件 + 关键点）
- 验证结果（复现前后对比）
- 风险与后续观察点
- planning 文件同步结果
```

---

## 推荐用法（补充）

- 遇到线上/联调问题优先使用“模版四”。

## 模版五：需求评审 / 方案设计（开工前控范围）

```text
你现在负责简擎需求评审与方案设计，请先评审再编码：

1) 读取上下文
- AGENTS.md
- 前后端三文件（task_plan.md / findings.md / progress.md）

2) 需求澄清（先问清再做）
- 业务目标是什么（为什么做）
- 成功标准是什么（如何算完成）
- 范围边界是什么（做什么/不做什么）
- 涉及哪些角色与权限
- 是否跨前后端、是否影响现有接口

3) 输出评审结论
- 当前理解（1段）
- 需求拆分（1~3 个最小闭环）
- 风险点（数据、权限、兼容性、性能）
- 待确认问题（必须确认项）

4) 方案设计约束
- 优先最小闭环，避免过度设计
- 命名语义直白，可读性优先
- 接口方法仅允许 GET/POST，不新增 PUT/DELETE
- 关键分支/边界处理写必要注释
- 跨前后端变更需同步两端 planning 文件

5) 交付物
- 技术方案（后端/前端/接口/数据影响）
- 实施步骤（按顺序）
- 验证方案（功能验证 + 构建验证）
- 回滚与兜底策略（如失败如何恢复）

6) 执行开关
- 先只输出“评审 + 方案”，等待我回复“开始实现”后再编码。
```

---

## 推荐用法（再补充）

- 新功能开工前优先使用“模版五”，先控范围再实施。

## 模版六：数据库变更（DDL / 迁移脚本）

```text
你现在负责简擎数据库变更任务，请按“先评审、后落库、可回滚”执行：

1) 读取上下文
- AGENTS.md
- 前后端三文件（task_plan.md / findings.md / progress.md）
- 现有 SQL：初始化脚本与 patch 脚本目录

2) 变更评审
- 变更类型：新增表 / 加字段 / 索引调整 / 数据修复
- 影响范围：后端实体/Mapper/服务、前端字段展示、历史数据兼容
- 风险评估：锁表风险、全表扫描风险、数据丢失风险

3) 设计约束
- 优先增量 patch，不直接破坏初始化脚本语义
- DDL 与数据修复脚本拆分清晰
- 命名与注释直白，可读性优先
- 兼容旧库：能补丁升级，不要求重置数据库

4) 实施顺序
- 先给 SQL 方案与回滚方案
- 再落地 `sql/patch/*.sql`（必要时同步 `sql/jianqing-init-v0.1.sql`）
- 同步后端代码（实体/Mapper/查询）
- 若影响前端展示，同步前端字段与交互

5) 验证要求
- 语法与执行顺序自检（建表/加列/索引/数据修复）
- 后端编译验证：mvn -DskipTests compile && mvn test
- 前端变更时执行：npm run build
- 接口方法仍需满足 GET/POST 约束

6) 输出格式
- 变更摘要（做了什么、为什么）
- SQL 清单（初始化/patch 分别列出）
- 兼容性说明（新库/旧库）
- 回滚方案
- planning 文件同步结果
```

---

## 推荐用法（最终补充）

- 涉及表结构或历史数据修复时优先使用“模版六”。

## 模版七：权限模型变更（RBAC 专用）

```text
你现在负责简擎 RBAC 权限模型变更，请按“先盘点、再改造、强回归”执行：

1) 读取上下文
- AGENTS.md
- 前后端三文件（task_plan.md / findings.md / progress.md）
- 权限相关代码与 SQL（用户/角色/菜单/角色菜单/用户角色）

2) 变更盘点
- 变更目标：新增权限、调整权限粒度、修复越权/漏权
- 影响对象：菜单可见性、按钮权限、接口权限、路由守卫
- 影响角色：管理员、普通角色、自定义角色

3) 设计约束
- 保持“菜单可见性”与“按钮权限”语义分离
- 接口鉴权与前端显隐要一致，避免前端放开但后端未放开
- 接口方法仅允许 GET/POST，不新增 PUT/DELETE
- 命名直白，权限标识统一规范（如 `system:user:edit`）

4) 实施顺序
- 先列权限矩阵（角色 -> 菜单/按钮/接口）
- 再改后端权限判定与查询链路
- 同步前端路由守卫与按钮显隐
- 如需历史兼容，补 `sql/patch/*.sql` 增量修复

5) 验证要求
- 角色维度回归：至少验证“有权限/无权限”两组场景
- 菜单与按钮联动验证：可见性与可操作性一致
- 后端编译验证：mvn -DskipTests compile && mvn test
- 前端构建验证：npm run build
- 方法约束扫描：确认无 PUT/DELETE 新增

6) 输出格式
- 变更前后权限矩阵
- 修改清单（后端/前端/SQL）
- 验证结果（按角色列出）
- 风险与回滚方案
- planning 文件同步结果
```

---

## 推荐用法（权限专项）

- 涉及菜单、按钮、角色分配、越权修复时优先使用“模版七”。
