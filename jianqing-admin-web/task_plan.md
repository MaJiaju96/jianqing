## Decisions Made
| Decision | Rationale |
|----------|-----------|
| 前端目录与后端平级 | 便于独立启动、构建、部署 |
| 前后端统一放置在 `code/jianqing` 工作区 | 便于单目录开启跨端协作开发 |
| 前端默认纯 JavaScript | 满足当前开发偏好，降低改动成本 |
| UI 采用玻璃态 + 渐变风格 | 避免传统后台“死板感” |
| 关键/复杂逻辑必须写注释 | 提高团队协作与后续维护效率 |
| 前端实现坚持极简可读 | 保证低学习成本和快速接手能力 |
| 兼容后续多前端分支演进 | 为 `TS` 分支与 `Vue2` 分支预留迁移空间 |
| 前后端接口方法统一为 `GET` / `POST` | 降低联调歧义，避免 `PUT` / `DELETE` 方法分歧 |
| 列表页统一分页交互 | 降低长列表认知负担，提升信息检索效率 |
| 菜单页增加类型筛选与层级信息 | 强化菜单/按钮权限边界与结构可读性 |
| 前端公共约定收敛到规范文档与常量模块 | 降低 AI/人工协作时的隐式约定理解成本 |

## Sync Rules
- 前后端均维护独立 `task_plan.md / findings.md / progress.md`。
- 发生跨项目变更时，两个目录的计划文件都要同步记录。
- 注释执行规则：仅在关键逻辑/复杂流程处添加必要注释，避免无意义注释噪音。
- 前端编码原则：优先简单直接实现，避免复杂抽象。
- 协作原则：命名与结构保持清晰，确保 AI 与开发者都可快速理解。
- 接口联调规则：前端 API 调用仅使用 `GET` 与 `POST`，不使用 `PUT` / `DELETE`。
- 可读性规则：优先通过常量、函数命名与结构收敛提升可读性，避免用大量解释性注释替代清晰代码。
