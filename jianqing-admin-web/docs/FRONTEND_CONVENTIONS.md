# 简擎前端开发规范

本文档用于约束 `jianqing-admin-web` 的可读性、协作方式与 AI 辅助开发行为。

## 1. 总原则

- 优先简单直接实现，避免为了抽象而抽象。
- 先保证页面可读、可维护，再考虑技巧型优化。
- 所有新增代码默认面向“人类开发者 + AI”共同阅读。

## 2. 注释规则

- 不要求每个函数都写注释，避免注释噪音。
- 以下场景必须补充必要注释：
  - 关键分支：例如登录态判断、权限分流、异常兜底。
  - 复杂流程：例如递归树过滤、批量勾选回填、状态同步。
  - 边界处理：例如空值过滤、401 失效跳转、筛选条件兼容。
- 注释只解释“为什么这样做”或“这里的约束是什么”，不要复述字面代码。

## 3. 常量与魔法值

- 禁止在多个页面重复散落业务魔法值。
- 以下内容优先提取为常量：
  - 接口成功码。
  - 分页大小。
  - 状态值（启用/禁用）。
  - 菜单类型值（目录/菜单/按钮）。
  - 特殊角色标识（如 `super_admin`）。
- 当前公共常量统一收敛在 `src/constants/app.js`。

## 4. 视图层约定

- 列表页优先保持统一结构：筛选区、表格区、分页区。
- 创建/编辑共用表单时，优先抽公共校验，再保留少量分支差异。
- 权限树、菜单树这类层级数据，优先保证结构可读性，不为了省几行代码牺牲理解成本。
- 复杂 UI 行为优先写成语义清晰的小函数，不堆叠匿名表达式。

## 5. API 层约定

- `src/api` 只负责接口语义表达，不混入页面状态。
- 响应解包、错误提示、401 处理统一在 `src/api/http.js` 收口。
- 查询参数序列化、接口成功码等通用约定必须写在公共层，避免页面重复猜测。
- 前后端联调仅使用 `GET` / `POST`，不新增 `PUT` / `DELETE`。

## 6. Store 与权限约定

- `authStore` 只持久化最小必要状态，避免把易过期数据长期写入本地存储。
- `profile` 作为权限判断依据，由登录后或路由守卫按需拉取。
- 权限判断统一走 `src/utils/permission.js`，不要在页面里散落重复权限表达式。
- 页面内批量权限显隐优先复用 `src/composables/usePermissions.js`。

## 7. 命名与结构约定

- 文件名保持语义直白：页面以 `*View.vue` 结尾。
- 函数名优先使用动词短语：`loadData`、`handleSubmit`、`openEdit`。
- 状态变量保持一致风格：`dialogVisible`、`pageNo`、`pageSize`、`statusFilter`。
- 新增目录前先判断是否真的需要分层，避免过度拆分。

## 8. AI 协作约定

- AI 修改前优先复用现有模式，不主动引入新风格。
- AI 新增逻辑时必须遵守本文注释规则和常量收敛规则。
- 涉及接口、权限、路由、主题等公共约定变更时，同步更新：
  - `README.md`
  - `findings.md`
  - `progress.md`
  - 本规范文档
- 若只是可读性优化，应避免改变业务行为和接口契约。
